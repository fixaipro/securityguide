True Database Isolation

Each customer gets their own PostgreSQL database—not filtered tables, not row-level security. Complete architectural isolation.

📊 Google Cloud SQL Instance (europe-west2)
  • tenant_rrlondon_db → RR London (Complete isolated database)
  • tenant_smartwheels_db → Smartwheels (Complete isolated database)
  • tenant_premier_db → Premier Auto (Complete isolated database)
  • shared_lookups_db → Work statuses, service codes (Shared only)
Critical Point: Cross-tenant queries are architecturally impossible. Even if our application had a security vulnerability, attackers cannot access data from other tenants because it exists in a completely separate database.

8-Layer Security Stack

Defense-in-depth architecture ensures multiple independent protections

8 Output Validation

Filters VINs, API keys, and sensitive data patterns before response delivery. Regex-based blocking prevents data leakage even if AI misbehaves.

7 AI Processing (Third-Party AI Service)

Sandboxed AI engine with no network access, no database credentials, no direct API access. Receives only sanitized prompts and pre-retrieved data. Enterprise agreement prohibits training on customer data.

6 Application Logic (Cloud Run)

Container isolation with input sanitization, rate limiting, and automatic scaling. Each request runs in isolated environment.

5 ORM Security (SQLAlchemy)

Automatic parameterized queries prevent SQL injection. Type validation rejects malformed input before reaching database.

4 Database Access (Cloud SQL Proxy)

IAM-based authentication with automatic credential rotation. All connections encrypted via TLS 1.3.

3 Database Server (Cloud SQL)

Private IP only, AES-256 encryption at rest, automated backups. 400-day audit log retention.

2 Network Security (VPC + Firewall)

Virtual Private Cloud with deny-all inbound rules. Database accessible only via authorized Cloud Run services.

1 DDoS Protection (Cloud Armor)

Edge protection with automatic DDoS mitigation, rate limiting, and geographic filtering.

Result: An attacker would need to bypass 8 consecutive security layers to access your API data. Each layer has independent controls.

SQLAlchemy ORM: Automatic SQL Injection Protection

All database queries use parameterization—user input is always treated as data, never as executable code

🛡️

Parameterized Queries

All queries automatically use parameters. SQL injection is impossible because user input never becomes SQL code.

✓

Type Validation

Data types enforced at ORM level. Invalid input (wrong type, format, length) rejected before reaching database.

🚫

No String Concatenation

Zero raw SQL strings. All queries built through ORM methods that automatically handle escaping.

Example: How It Protects Against SQL Injection

# User provides: registration = "AB12 CDE" vehicle = db.query(Vehicle).filter(Vehicle.registration == user_input).first() # SQLAlchemy generates SAFE query: # SELECT * FROM vehicles WHERE registration = $1 # Parameters: ['AB12 CDE'] ← Treated as data, not code # Even with malicious input: "AB12'; DROP TABLE vehicles; --" # SQLAlchemy safely escapes it as a string search # SELECT * FROM vehicles WHERE registration = $1 # Parameters: ["AB12'; DROP TABLE vehicles; --"] ← Harmless

24-Hour Data Lifecycle

Automated deletion ensures compliance with data minimization principles

Hour 0: Data Retrieved

Customer calls asking about MOT. Backend queries your API and stores result in tenant's isolated database with timestamp.

Hours 0-24: Cache Available

If same customer calls again within 24 hours, we use cached data (reduces API calls to your service).

Hour 24: Automatic Deletion

Cloud Function runs at 2:00 AM UTC daily. Deletes all records with retrieved_at > 24 hours ago.

Audit Trail: Proof of Deletion

Cloud Logging captures deletion events with timestamps, record counts, and database names. 400-day retention for compliance verification.

# Cloud Function: Runs daily at 2:00 AM UTC def purge_expired_data(): for tenant_db in all_tenant_databases: # Delete vehicle data older than 24 hours result = tenant_db.execute(""" DELETE FROM vehicle_cache WHERE retrieved_at < NOW() - INTERVAL '24 hours' RETURNING id """) # Log deletion for audit trail log_to_cloud_logging({ 'database': tenant_db.name, 'deleted_count': result.rowcount, 'timestamp': datetime.utcnow() })

What We DON'T Do

Clear boundaries on data usage and business model

❌ No Dataset Building

We don't create derivative databases or aggregate vehicle data across customers. Data is used transiently and deleted within 24 hours.

❌ No AI Training

Your vehicle data never enters any machine learning training pipeline. We use pre-trained models with zero data retention.

❌ No Data Sales

We sell automation subscriptions (£99-299/month per garage), not data. Revenue comes from services, not information resale.

❌ No Cross-Tenant Analysis

Database isolation makes aggregation across tenants architecturally impossible. Each customer's data is completely separate.

❌ No Long-Term Storage

No "just in case" data hoarding. Automated deletion enforces 24-hour maximum retention with audit trails.

❌ No Third-Party Sharing

Vehicle data never leaves our GCP infrastructure. No sharing with advertisers, data brokers, or analytics services.
